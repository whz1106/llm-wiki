# LLM Performance Testing

## 定义

[[llm-performance-testing]] 指针对大语言模型或其上层推理服务进行的性能评估活动。它区别于传统接口性能测试的关键点在于：测试对象往往是流式输出、长耗时、强依赖输入规模和推理架构的系统。

## 指标清单

大模型性能测试不能只看单一“快不快”，更应同时覆盖效果、性能、成本、安全和业务结果。实际落地时，可以按以下几类组织指标。

### 1. 效果指标

#### 准确性

- 回答是否正确
- 适用于问答、分类、抽取、代码生成等任务

#### 相关性

- 输出是否切题
- 是否真正回应用户问题或业务目标

#### 完整性

- 是否覆盖任务要求中的关键点
- 是否存在明显遗漏或答非所问

#### 一致性

- 同样输入多次提问时，结果是否稳定
- 多轮对话中前后结论是否矛盾

#### 事实性 / 幻觉率

- 是否编造事实、来源、链接、接口或代码能力
- 在知识问答、RAG、专业咨询场景中尤为关键

#### 推理质量

- 多步推理是否连贯
- 数学、逻辑、条件约束处理是否可靠

### 2. 指令与格式遵循指标

#### 指令遵循率

- 是否按照用户要求完成任务
- 是否满足长度、语气、角色、约束条件等要求

#### 格式正确率

- 是否按要求输出 JSON、表格、Markdown、代码块等结构
- 结构化输出是否可解析、字段是否齐全

#### 可执行性

- 代码、SQL、Shell、配置输出是否可直接运行
- 是否满足语法、依赖和接口约束

### 3. 鲁棒性指标

#### 改写鲁棒性

- 问法变化后，结果是否明显波动

#### 噪声容忍度

- 在口语化输入、错别字、冗余上下文下表现是否稳定

#### 边界样本稳定性

- 长上下文、歧义问题、冲突信息、极端输入下是否退化明显

### 4. 性能指标

#### 首 token 耗时

- 从请求发出到开始输出首个 token 的时间

#### 吐字率

- 输出 token/s
- 反映 decode 阶段的持续生成能力

#### 总响应时间

- 从请求发出到完整结果返回的总耗时

#### QPM / 吞吐量

- 单位时间内系统可处理的问题数
- 常与并发压测一起观察

#### 输入 / 输出 token

- 需要与延迟、吞吐量和成本一起看
- 是分析性能波动的基础上下文

#### 成功率与回答完整性

- 请求是否成功返回
- 输出是否中断、截断或提前结束

#### 并发稳定性

- 多用户同时请求时，延迟、失败率和吐字率是否恶化

### 5. 成本指标

#### Token 成本

- 输入 token 成本
- 输出 token 成本

#### 单任务成本

- 完成一次实际业务任务的总成本

#### 单位质量成本

- 在达到预期质量前提下的成本
- 用于比较不同模型、参数和提示词方案的性价比

### 6. 安全与风险指标

#### 有害内容风险

- 是否生成违法、暴力、仇恨、自残等高风险内容

#### 越狱成功率

- 是否容易被恶意提示绕过约束

#### 提示注入抵抗能力

- 在工具调用、RAG、Agent 场景中是否会被外部文本操控

#### 隐私泄露风险

- 是否输出敏感信息、系统提示或不应暴露的上下文

#### 合规性

- 在医疗、法律、金融等高风险领域是否守住边界

### 7. 业务结果指标

#### 任务完成率

- 是否真正完成用户目标，而不是只生成表面合理的回答

#### 用户满意度

- 用户评分
- 点赞率
- 复用率

#### 运营与工程结果

- 转人工率
- 工单关闭率
- 单测通过率
- 编译通过率
- 工具调用成功率

## 最常用的一组核心指标

- 首 token 耗时
- 吐字率（输出 token/s）
- QPM
- 输入 token 数量
- 输出 token 数量
- 成功率与回答完整性
- 指令遵循率
- 幻觉率
- 单任务成本
- 安全违规率
- 任务完成率

## 为什么它与普通性能测试不同

- 用户是按“先返回得快不快、持续输出稳不稳”感知体验的，而不是只看总响应时间。
- 同一个模型在不同 token 长度、不同桶位、不同参数下，性能差异可能很大。
- 很多优化只作用于某个推理阶段，因此不能只看单一平均值。

## 常用评测方法

- 自动评测：EM、F1、ROUGE、BLEU、pass@k、JSON Schema 校验
- 人工评测：适合评估相关性、完整性、可用性和表达质量
- 模型裁判：适合大规模初筛，但需要注意偏差
- A/B 测试：用真实用户行为和业务指标验证模型表现
