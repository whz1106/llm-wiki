# 通往专家之路，三万字长文详解大模型性能测试

## 核心观点

- 大模型性能测试不能只看传统的响应时间和 QPS，更要关注 [[llm-performance-testing]] 中的首 token、吐字率和 QPM。
- 性能数据必须和输入 token、输出 token、模型桶位、推理参数一起看，否则很容易得出错误结论。
- [[prefill-decode]] 的拆分是理解指标的关键：首 token 更接近 prefill 表现，吐字率更接近 decode 表现。
- 架构优化不能只测“快没快”，还要测“值不值得”，因为量化、并行、缓存等优化都可能影响精度或稳定性。
- 测试人员的核心能力不是会写压测脚本，而是能判断某个指标变化是否符合理论预期。

## 关键内容

### 一般性能测试

文章先建立了大模型性能测试的基本方法：

- 对齐 `token`、输入输出长度、分桶策略
- 针对流式输出接口做压测
- 自定义上报首 token、吐字率、QPM、输入输出 token
- 用二次处理脚本清洗压测结果

### 进阶架构测试

后半篇转向推理架构优化的验证，包括：

- [[prefill-decode]]
- KV cache 命中与路由策略
- [[pd-separation]]
- MTP、量化、TP/DP/PP/EP/MOE

这里最重要的不是术语本身，而是“每个优化理论上应该影响哪个指标”。

## 可归档的判断框架

可以把文章压缩成一个实用检查表：

1. 这次测试的是哪个桶、哪个模型、哪个参数组合？
2. 关注的优化作用在 prefill 还是 decode？
3. 指标变化是否符合该优化的理论作用面？
4. 是否同时验证了成功率、完整性和效果退化？
5. 数据是否足够接近线上真实流量结构？
