## 恢复版内容提纲

### 1. 前言

文章定位为“大模型性能测试”的系统化长文，目标不是讲如何用大模型生成测试用例，而是讲测试人员如何真正评估大模型系统的性能、架构优化和测试策略。

### 2. 专业名词对齐

这部分先统一术语，主要包括：

- `token` 是模型计算和计费的基本单位，不等于“字数”。
- 模型通常会声明支持的输入长度和输出长度，例如 `32K/16K`。
- 测试数据需要先按 token 数量分组，再对不同规模分别做性能评估。
- 真实线上场景里，同一模型往往会按输入规模或场景进行“分桶”。

作者在这里强调一个核心认知：测试人员测的不是抽象的“AI 能力”，而是特定 token 规模、特定桶位、特定参数下的系统表现。

### 3. 大模型与常规产品性能测试的差异点

文章把差异拆成两块：

- 指标不同
- 发压方式不同

大模型场景里的关键性能指标包括：

- 吐字率：平均输出 token 速率
- 首 token 耗时：从用户发问到返回第一个 token 的时间
- QPM：每分钟请求数
- 平均输入 token 数量
- 平均输出 token 数量

作者指出，普通 HTTP 接口的平均响应时间和 QPS 不足以描述大模型体验，因为用户是按流式输出感知“模型快不快”的。

### 4. 流式接口压测与指标计算

文章使用 OpenAI SDK 兼容接口说明如何压测流式输出模型，重点包括：

- 区分思考包、答案包和尾包
- 思考模型需要分别关注首思考包和首答案包
- `locust` 默认指标不够，需要自定义上报
- 压测完成后，需要对导出的 CSV 做二次处理，生成 QPM、首 token、吐字率等结果

文章中给出了 Python 版的压测逻辑和 CSV 汇总脚本，用于把原始压测结果转成更适合大模型分析的指标表。

### 5. 性能测试进阶

从这部分开始，文章转向“推理框架优化测试”而不只是“普通压测”。

作者先强调：

- 模型本身和推理框架是两回事
- 同一个模型，因为推理框架策略不同，性能表现会差很多
- 测试人员需要理解常见加速方案，否则无法判断结果是否合理

随后文章解释了几个重要概念：

- 大模型生成 token 的基本过程
- sampling、top-k、top-p、temperature 等参数为何会影响稳定性和评估
- 推理优化不仅影响速度，也可能影响效果，因此要同步做精度测试

### 6. Prefill 与 Decode

这是全文非常关键的一段。

- `prefill` 主要负责首 token 以及 KV 计算
- `decode` 主要负责后续 token 的持续生成
- 首 token 主要反映 prefill 阶段表现
- 吐字率主要反映 decode 阶段表现

文章据此引出测试思路：

- 如果在优化 KV cache，就更该关注首 token 和缓存命中收益
- 如果在优化 decode 侧，就更该关注吐字率
- 测试脚本不能只会“发请求”，还要能刻意命中或规避缓存

### 7. KV Cache 测试

作者把 KV cache 的测试拆得很具体：

- 不想命中缓存时，在输入前缀加 `uuid`
- 想验证缓存收益时，去掉 `uuid`
- 为了缩短测试时长，可以限制 `max_tokens`
- 基准测试可以用同样数据发两次，比较第二次的命中收益
- 更真实的评估要基于按时间顺序采集的线上数据

文中也特别提醒了几个现实问题：

- 缓存命中率依赖路由策略
- 命中的前缀长度并不固定
- 缓存会淘汰，长文本 KV cache 的资源占用很高

### 8. P/D 分离

文章把 `P/D 分离` 作为一类重要架构优化方案：

- `P` 指 prefill
- `D` 指 decode
- 分离后可以分别优化，但会显著增加架构复杂度

测试人员在这里要覆盖的不只是性能本身，还包括：

- P/D 之间的通信可靠性
- 故障时的降级逻辑
- 不同 `NP/ND` 配比的性能差异
- 多 group 分桶策略
- 降级成不分离方案后的性能回退情况
- 不同阶段算力打满后对指标的影响

文章明确指出：如果做了只影响 decode 的优化，却让首 token 指标变化明显，就应优先怀疑环境、脚本或配置异常。

### 9. 精度、量化与 MTP

进阶部分还讨论了多种加速方案对应的测试关注点：

- 降低存储精度（如 `fp32 -> fp16 -> fp8/int4`）会带来性能提升，但可能损伤效果
- `MTP`（多头预测）主要提升吐字率，不应显著改善首 token
- 这类优化需要让研发侧提供可观测指标，例如 MTP 小模型的准确率

这里的核心原则是：性能收益要和理论机制匹配，否则测试结果本身就值得怀疑。

### 10. TP / DP / PP / EP / MOE

文章用测试人员能理解的程度解释并行策略：

- `TP`：张量并行
- `DP`：数据并行
- `PP`：流水线并行
- `EP`：专家并行
- `MOE`：混合专家结构

作者特别提到：

- P/D 分离通常对首 token 不友好，但可能有更好的吐字率
- EP 是当前主流方向之一
- 测试人员不必成为算法专家，但必须知道这些策略理论上会影响哪个指标

### 11. 其他压测场景

后半段还覆盖了两个很实用的场景：

- 逐步加压：避免一开始瞬时灌满并发，导致首 token 指标失真
- 模拟大集群压测：在模型层做 mock，只测接入层和路由层的大规模 QPM

对应工具上，文章提到：

- `locust` 适合一般压测，但原生并发能力有限
- `boomer` 作为 golang worker，适合更高压场景

### 12. 尾声

文章最后收束为一个很明确的立场：

- 测试人员不一定要成为算法工程师
- 但必须知道自己在测什么、指标为什么变、哪些变化合理、哪些变化不合理
- 真正困难的部分不是写压测脚本，而是建立技术判断力
